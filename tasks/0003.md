## Context

The Database Manager is the most foundational core service. Every other service and every feature depends on it. It is responsible for:

- Creating and opening SQLite databases (core + per-feature) with WAL mode
- Running migrations and tracking which have been applied
- Providing database instances to features via `FeatureContext`
- Managing database file locations on disk

The full core database schema from `specs/ARCHITECTURE.md` is large (features, scheduled_tasks, script_executions, event_log, settings, credentials, etc.). This task creates only the **migration infrastructure and the minimum tables needed for Phase 1**: `features` and `settings`. Later tasks add their own core migrations as each service is implemented (scheduler, script engine, event bus, etc.).

## What This Task Does NOT Do

- Does NOT implement the Feature Registry, Settings Manager, or any other core service ã those are separate tasks that consume the Database Manager
- Does NOT create the full core schema ã only the `features`, `settings`, and `migrations` tables
- Does NOT implement backup/restore
- Does NOT write feature-specific database logic

### Part B: Implement the DatabaseManager class

Create `src/core/bun/database-manager.ts` with the following API:

```typescript
import { Database } from "bun:sqlite";

export class DatabaseManager {
  private dataDir: string;
  private databases: Map<string, Database>;
  private coreDb: Database | null;

  constructor(dataDir: string);

  getCoreDatabase(): Database;
  getFeatureDatabase(featureId: string): Database;
  closeAll(): void;
}
```

**Key behaviors:**

1. **`constructor(dataDir)`** ã stores the data directory path. Creates `dataDir` and `dataDir/features/` directories if they don't exist (`mkdir -p` equivalent).

2. **`getCoreDatabase()`** ã opens (or returns cached) `core.db` in the data directory. On first open:
   - Enables WAL mode: `PRAGMA journal_mode=WAL`
   - Enables foreign keys: `PRAGMA foreign_keys=ON`
   - Bootstraps the `migrations` table (direct CREATE IF NOT EXISTS ã this table cannot be migrated because it tracks migrations)
   - Runs pending core migrations

3. **`getFeatureDatabase(featureId)`** ã opens (or returns cached) `features/{featureId}.db`. On first open:
   - Enables WAL mode
   - Enables foreign keys
   - Bootstraps the `migrations` table in the feature DB
   - Does NOT run migrations ã that's the Feature Registry's job when activating a feature

4. **`closeAll()`** ã closes all open database connections. Called during app shutdown.

### Part C: Implement the migration runner

Add a migration runner as a private method (or separate internal module) used by the Database Manager.

**`src/core/bun/migration-runner.ts`**

```typescript
import { Database } from "bun:sqlite";
import type { Migration } from "@core/types";

export function bootstrapMigrationsTable(db: Database): void;
export function runMigrations(
  db: Database,
  featureId: string,
  migrations: Migration[],
): { applied: string[]; skipped: string[] };
export function getAppliedMigrations(
  db: Database,
  featureId: string,
): string[];
```

> **No separate `MigrationDefinition` type.** The `Migration` interface from `@core/types/feature.ts` (defined in task 0002) has the exact shape needed: `{ version, name, up, down? }`. Reuse it everywhere ã don't create a parallel type.

**Key behaviors:**

1. **`bootstrapMigrationsTable(db)`** ã creates the `migrations` table if it doesn't exist:
   ```sql
   CREATE TABLE IF NOT EXISTS migrations (
       feature_id TEXT NOT NULL,
       version    TEXT NOT NULL,
       name       TEXT NOT NULL,
       applied_at TEXT NOT NULL,
       PRIMARY KEY (feature_id, version)
   );
   ```

2. **`runMigrations(db, featureId, migrations)`** ã for each migration in order:
   - Check if `(featureId, version)` already exists in `migrations`
   - If not: execute the `up` SQL, then insert a row into `migrations`
   - Each migration runs in a transaction (rollback on failure)
   - Returns which migrations were applied and which were skipped

3. **`getAppliedMigrations(db, featureId)`** ã returns the list of version strings already applied.

## Notes

- **Bun has `bun:sqlite` built in.** No additional SQLite package needed. Use `import { Database } from "bun:sqlite"`.
- **The `migrations` table bootstraps itself** with `CREATE TABLE IF NOT EXISTS` ã it cannot use the migration system it enables. Everything else goes through migrations.
- **Only `features` and `settings` tables in the initial migration.** The remaining core tables (`scheduled_tasks`, `script_executions`, `event_log`, `scripts`, `credentials`, etc.) are added by later tasks as each core service is implemented. Each service brings its own migration.
- **Feature migrations are NOT run by the Database Manager.** It only creates the database file and the `migrations` table. The Feature Registry (a later task) is responsible for running a feature's migrations during `install`/`activate`.
- **The `data/` directory is in `.gitignore`** (added in task 0001).
- **Read `specs/TESTING.md` before writing tests.** It defines file placement (co-located), naming (`*.test.ts`), structure (`describe`/`test`), assertion style, and the temp directory pattern for database tests.
