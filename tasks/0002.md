## Context

With the project cleaned up and restructured (task 0001), the next step is to install the runtime dependencies the core shell needs and to define the TypeScript type system that every core service and feature will build against.

The type system is the foundational contract of the entire app. It defines what a feature is (`FeatureDefinition`), what the core provides to features (`FeatureContext`), and how features communicate (events, actions, queries). Getting this right before writing any implementation code prevents rework across every service and feature.

This task also installs the minimum set of dependencies required for Phase 1 work — no more than what's needed.

## What This Task Does NOT Do

- Does NOT implement any core services (database manager, event bus, scheduler, etc.)
- Does NOT set up shadcn/ui components — that's a separate task
- Does NOT create the dashboard grid — that's a separate task
- Does NOT write any test files — the types are verified by the TypeScript compiler. But it does set up the test infrastructure (Vitest config, test scripts) so that task 0003+ can write tests immediately.

#### C2. `src/core/types/feature.ts`

The `FeatureDefinition` interface — the contract every feature must implement:

```typescript
import type { EventMap, ActionMap, QueryMap } from "./common";
import type { FeatureContext, FeatureLifecycleContext } from "./context";

export interface Migration {
  version: string;
  name: string;
  up: string;
  down?: string;
}

export interface EventDeclaration {
  description?: string;
  payload?: Record<string, string>;
}

export interface ActionDeclaration {
  description?: string;
  params?: Record<string, string>;
  result?: Record<string, string>;
}

export interface QueryDeclaration {
  description?: string;
  params?: Record<string, string>;
  result?: string | Record<string, string>;
}

// Schema fields (payload, params, result) are optional metadata for documentation,
// command palette hints, and future validation. Runtime type safety comes from
// the generic type parameters on FeatureDefinition, not from these schemas.

export type EventDeclarations<T extends EventMap> = {
  [K in keyof T]: EventDeclaration;
};

export type ActionDeclarations<T extends ActionMap> = {
  [K in keyof T]: ActionDeclaration;
};

export type QueryDeclarations<T extends QueryMap> = {
  [K in keyof T]: QueryDeclaration;
};

export interface Permission {
  type: "network" | "notifications" | "clipboard" | "filesystem";
  reason: string;
}

export interface ScheduledTaskDeclaration {
  id: string;
  defaultSchedule: {
    type: "cron" | "interval";
    value: string | number;
  };
  description?: string;
}

export type WidgetSize = "small" | "medium" | "wide" | "full-width";

export interface WidgetDeclaration {
  id: string;
  sizes: WidgetSize[];
  name?: string;
  description?: string;
}

export interface CommandDeclaration {
  id: string;
  label: string;
  params?: string[];
  description?: string;
}

export interface FeatureManifest<
  TEvents extends EventMap = EventMap,
  TActions extends ActionMap = ActionMap,
  TQueries extends QueryMap = QueryMap,
> {
  events: EventDeclarations<TEvents>;
  actions: ActionDeclarations<TActions>;
  queries: QueryDeclarations<TQueries>;
  permissions: Permission[];
  scheduledTasks: ScheduledTaskDeclaration[];
  widgets: WidgetDeclaration[];
  commands: CommandDeclaration[];
}

export interface FeatureDefinition<
  TEvents extends EventMap = EventMap,
  TActions extends ActionMap = ActionMap,
  TQueries extends QueryMap = QueryMap,
> {
  id: string;
  name: string;
  version: string;

  install(ctx: FeatureLifecycleContext): Promise<void>;
  activate(ctx: FeatureContext<TEvents, TActions, TQueries>): Promise<void>;
  deactivate(): Promise<void>;
  uninstall(ctx: FeatureLifecycleContext): Promise<void>;

  migrations: Migration[];
  manifest: FeatureManifest<TEvents, TActions, TQueries>;
}
```

#### C3. `src/core/types/context.ts`

The `FeatureContext` interface — what the core provides to each feature at activation:

```typescript
import type { Database } from "bun:sqlite";
import type { EventMap, ActionMap, QueryMap } from "./common";

export interface ActionMeta {
  executionId: string;
  correlationId: string;
  retriedCount: number;
}

export interface ScopedLogger {
  info(message: string, ...args: unknown[]): void;
  warn(message: string, ...args: unknown[]): void;
  error(message: string, ...args: unknown[]): void;
  debug(message: string, ...args: unknown[]): void;
}

export interface FeatureLifecycleContext {
  db: Database;
  log: ScopedLogger;
}

export interface FeatureContext<
  TEvents extends EventMap = EventMap,
  TActions extends ActionMap = ActionMap,
  TQueries extends QueryMap = QueryMap,
> {
  db: Database;

  events: {
    emit<K extends keyof TEvents>(event: K, payload: TEvents[K]): void;
  };

  actions: {
    handle<K extends keyof TActions>(
      action: K,
      handler: (
        params: TActions[K]["params"],
        meta: ActionMeta,
      ) => Promise<TActions[K]["result"]>,
    ): void;
  };

  queries: {
    handle<K extends keyof TQueries>(
      query: K,
      handler: (
        params: TQueries[K]["params"],
      ) => Promise<TQueries[K]["result"]>,
    ): void;
  };

  subscribe(
    event: string,
    handler: (payload: unknown) => Promise<void>,
  ): void;

  query(
    feature: string,
    queryName: string,
    params: unknown,
  ): Promise<unknown>;

  scheduler: {
    register(taskId: string, handler: () => Promise<void>): void;
  };

  settings: {
    get<T>(key: string, defaultValue: T): T;
    set(key: string, value: unknown): Promise<void>;
  };

  log: ScopedLogger;
}
```

### Part D: Set up test infrastructure

#### D1. Add test scripts to `package.json`

```json
{
  "scripts": {
    "test": "bun test src/core src/features && vitest run",
    "test:core": "bun test src/core src/features",
    "test:ui": "vitest run",
    "test:watch": "bun test --watch src/core src/features",
    "test:ui:watch": "vitest"
  }
}
```

> **Why explicit paths?** `bun test` without arguments discovers all `*.test.{ts,tsx}` files project-wide, including UI test files that import from `vitest`. Those files would fail under Bun's test runner. Passing `src/core src/features` restricts discovery to Bun-side code.

#### D2. Create `vitest.config.ts`

Create at the project root:

```typescript
import { defineConfig } from "vitest/config";
import react from "@vitejs/plugin-react";
import path from "path";

export default defineConfig({
  plugins: [react()],
  test: {
    environment: "jsdom",
    include: [
      "src/shell/view/**/*.test.{ts,tsx}",
      "src/features/**/view/**/*.test.{ts,tsx}",
      "src/core/ui/**/*.test.{ts,tsx}",
    ],
    globals: true,
  },
  resolve: {
    alias: {
      "@core": path.resolve(__dirname, "src/core"),
      "@shell": path.resolve(__dirname, "src/shell"),
      "@features": path.resolve(__dirname, "src/features"),
    },
  },
});
```

## Notes

- The `bun:sqlite` `Database` type is used in `FeatureContext` and `FeatureLifecycleContext`. Bun provides this natively — no additional package needed.
- These types are intentionally permissive at this stage (e.g., `unknown` in maps). They will be tightened when concrete features are implemented.
- The `FeatureContext.query()` method uses `unknown` params/return because it crosses feature boundaries — the caller doesn't know the target feature's types. Type safety within a feature is enforced by the generic parameters on `FeatureDefinition`.
- shadcn/ui setup is deferred to a later task because it requires its own CLI initialization, Tailwind plugin configuration, and component decisions.
- **Testing infrastructure is set up here** (Vitest, RTL, test scripts) even though no test files are written. This ensures task 0003 can immediately write and run tests. See `specs/TESTING.md` for all conventions.
